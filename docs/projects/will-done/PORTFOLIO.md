# 📋 will-done — 바이브 코딩 포트폴리오

> **"흩어진 내 업무 시간을 커리어로 연결하는 타임 트래커"**  
> 코드 한 줄 직접 작성하지 않고, AI 에이전트만으로 풀스택 데스크톱 앱을 구축한 바이브 코딩 프로젝트

---

## 1. 프로젝트 개요

### 🎯 기획 배경

- 이직 준비 시 "그동안 뭘 했지?" 하고 기억이 나지 않는 문제
- 긴급 업무 발생 시 오늘 계획했던 일정이 무너지는 경험
- 기존 TODO 앱은 "시간"을 설계하지 않고 단순 나열에 그치는 한계

이러한 페인 포인트에서 출발하여, **업무 시간(will)을 설계**하고 **완료 기록(done)을 Brag Document로 변환**해주는 데스크톱 생산성 도구를 기획했습니다.

### 🛠️ 기술 스택

| Layer | Stack |
|---|---|
| **Frontend** | React 19, TypeScript, Vite 7, Tailwind CSS, shadcn/ui, Framer Motion, dnd-kit |
| **Backend** | Tauri v2, Rust, SQLite (sqlx 비동기), tokio |
| **AI** | Google Gemini API (Multi-Model Fallback Engine) |
| **CI/CD** | GitHub Actions (macOS/Windows 크로스 플랫폼 자동 빌드 & 릴리스) |
| **Auto Update** | Tauri Updater Plugin (바이너리 서명 검증) |

## 2. 핵심 기능 아키텍처

### ⏳ 지능형 타임 시프트 엔진 (Time-Shift Engine)

단순 TODO 앱과 차별화되는 핵심 비즈니스 로직입니다.

- **긴급 업무 대응**: 긴급 태스크 발생 시 현재 진행 중 태스크를 중단(PENDING) 및 분할하고, 이후 모든 일정을 수학적으로 계산하여 자동 시프트
- **언플러그드 타임 회피**: 점심시간, 정기 회의 등 고정 시간대를 설정하면 스케줄러가 자동으로 태스크 블록을 분할 배정
- **완료/연장 시 타임 시프트**: 조기 완료 시 이후 일정 당김(Pull up), 지연 완료 시 이후 일정 밀기(Push back)

### 🤖 AI 회고 및 Brag Document 자동화

- **성과 보고서 자동 생성**: 완료 태스크의 계획 메모와 리뷰 메모를 취합하여 마크다운 성과 보고서 생성
- **다중 모델 폴백(Fallback) 엔진**: Gemini API의 Rate Limit(429) 또는 서버 에러 발생 시 `로컬 캐시 → flash-lite → flash → pro` 순으로 자동 전환 재시도
- **다중 주기 지원**: 일간/주간/월간 단위 회고 생성 및 조회

### 🏢 워크스페이스 기반 컨텍스트 분리

- 코어 타임, AI 프롬프트 컨텍스트를 워크스페이스 단위로 완벽 격리
- 사이드 프로젝트와 본업을 섞임 없이 관리
- 워크스페이스별 고정 업무(루틴) 템플릿 관리 및 오늘의 루틴 추천

### 🔒 로컬 퍼스트 아키텍처

- 모든 데이터는 사용자 로컬의 SQLite에 저장
- 외부 서버 의존 없이 완벽한 데이터 주권 보장
- Tauri v2 기반 네이티브 퍼포먼스

## 3. 바이브 코딩 전략 — "코드를 작성하지 않고 소프트웨어를 만드는 법"

### 🧠 핵심 철학

이 프로젝트의 코드는 **100% AI 에이전트가 작성**했습니다. 저의 역할은 기획자(Product Owner)이자 AI 오케스트레이터(Orchestrator)로서, 각 AI 도구의 강점에 맞는 작업을 분배하고, AI가 올바른 방향으로 작업할 수 있도록 **구조화된 레퍼런스 문서를 설계**하는 데 집중했습니다.

### 🔧 AI 도구 멀티 에이전트 활용 전략

세 가지 AI 코딩 에이전트를 **적재적소에** 활용했습니다.

| AI 도구 | 역할 및 활용 시나리오 |
|---|---|
| **Gemini CLI** | 터미널 기반 CLI 에이전트. Rust 백엔드 로직, 데이터베이스 스키마 설계, Tauri 커맨드 구현 등 **백엔드 중심 작업**에 주로 활용. `GEMINI.md` 루트 룰 파일을 통해 프로젝트 컨텍스트를 자동 주입 |
| **oh-my-opencode** | 터미널 기반 에이전트. 빠른 버그 수정, 단일 파일 리팩토링, 간단한 기능 추가 등 **빠른 반복(Iteration) 작업**에 활용 |
| **Google Antigravity** | IDE 통합 에이전트. 다중 파일에 걸친 복잡한 프론트엔드 리팩토링, i18n 전면 적용, 컴포넌트 간 타입 호환성 해결 등 **넓은 컨텍스트가 필요한 작업**에 활용 |

### 📑 AI를 위한 레퍼런스 문서 시스템 설계

바이브 코딩의 성패는 **AI에게 얼마나 정확한 컨텍스트를 제공하느냐**에 달려 있습니다. 이 프로젝트에서는 `/docs/ai/` 디렉토리에 AI 전용 레퍼런스 문서를 체계적으로 구축했습니다.

#### `STRUCTURE.md` — 살아있는 프로젝트 지도 (240줄)

프로젝트의 **현재 상태**를 정의하는 핵심 문서입니다.

```
포함 내용:
├── 1. Directory Architecture     — 프론트엔드/백엔드 전체 파일 구조
├── 2. UI Layout Hierarchy        — 화면 위계 및 컴포넌트 트리
├── 3. Implemented Features & API — 구현 완료된 커맨드 및 기능 명세
└── 4. User Action Flow (A~N)     — 14개의 상세한 사용자 시나리오
```

**핵심 전략**: AI가 새 태스크를 시작할 때 이 문서를 먼저 읽게 함으로써, 기존 구현과 충돌하지 않는 코드를 생성하게 유도했습니다. 또한 AI가 작업을 완료할 때마다 이 문서를 **자동으로 업데이트**하도록 규칙을 설정하여, 문서와 코드가 항상 동기화된 상태를 유지했습니다.

#### `GUIDE.md` — 디자인 시스템 & 코딩 컨벤션

AI가 **일관된 품질**로 코드를 생성하도록 하는 가이드라인입니다.

```
포함 내용:
├── 1. Design System & Typography — 폰트 크기, 계층, 시각 요소 제한 규칙
├── 2. Frontend Guidelines        — 파일 분할 기준(250줄/5훅), 상태 관리 규칙
├── 3. React UI Testing           — Vitest + RTL 테스트 작성 기준
├── 4. Backend Guidelines         — Rust 모듈 아키텍처 분할 규칙
├── 5. Internationalization       — i18n 하드코딩 금지 원칙
└── 6. Documentation & Context    — 문서 업데이트 의무
```

#### `GEMINI.md` — 에이전트 하네스(Harness) 프로토콜

AI가 "말만 하고 행동하지 않는" 문제, 즉 **하네스(Harness) 문제**를 해결하기 위해 설계한 핵심 규칙 파일입니다.

## 4. 하네스(Harness) 문제 해결 — 바이브 코딩의 핵심 과제

### 🚨 문제 정의

바이브 코딩에서 가장 큰 도전은 AI 에이전트의 **하네스 문제(Harness Problem)**입니다.

> **하네스 문제**: AI 에이전트가 작업을 수행할 때, 목표를 이해했음에도 불구하고 실제 실행(코드 작성, 빌드, 커밋)으로 이어지지 않거나, 여러 작업을 동시에 처리하다가 혼란에 빠지는 현상

구체적으로 관찰된 증상들:

1. **배치 처리 경향**: LLM이 여러 태스크를 한꺼번에 처리하려다 중간에 맥락을 잃음
2. **검증 없는 진행**: 빌드가 깨진 상태에서 다음 태스크로 넘어감
3. **문서-코드 불일치**: 코드는 변경했지만 레퍼런스 문서 업데이트를 건너뜀
4. **불완전한 커밋**: 관련 없는 파일이 섞이거나, 커밋 단위가 비논리적

### ✅ 해결 전략: 3중 안전장치

#### 전략 1 — Strict Single-Tasking Loop (강제 단일 작업 루프)

```
PLANNING.md 기반 3-State 워크플로우:

1. [ ] To Do    → 대기 중인 작업
2. [~] In Progress → 현재 작업 중 (오직 하나만)
3. [x] Done     → 완료 및 검증됨

매 태스크마다:
  ① PLANNING.md에서 [ ] → [~] 변경 (코드 작성 금지)
  ② 해당 태스크만 코드 작성
  ③ cargo check / pnpm build 실행으로 검증
  ④ 통과 시 [~] → [x] 변경
  ⑤ 다음 태스크로 반복
```

LLM의 배치 처리 경향을 원천 차단하고, **한 번에 하나의 작업만** 수행하도록 강제했습니다.

#### 전략 2 — Self-Healing Verification (자기 치유 검증)

```
검증 실패 시 흐름:

  빌드/테스트 실패 → DO NOT PROCEED
      ↓
  에러 로그 분석
      ↓
  Root Cause 수정
      ↓
  재빌드/재테스트
      ↓
  통과할 때까지 반복
```

AI가 **빌드가 깨진 상태로 절대 다음 단계로 진행하지 않도록** 루프를 설계했습니다. 이를 통해 커밋 시점에는 항상 안정적인 코드 상태가 보장됩니다.

#### 전략 3 — Atomic Commit Protocol (원자적 커밋 프로토콜)

```
1. PLANNING.md의 모든 태스크가 [x]인지 확인
2. PLANNING.md 파일 삭제 (임시 파일 정리)
3. git add {관련_파일만} — 레이어별 분리 스테이징
4. git commit -m "{type}: {한국어 설명}" — 유형별 분리 커밋
5. git push
```

AI에게 **스테이징과 커밋의 논리적 단위**를 명확히 지시함으로써, `feat(fe):`, `feat(be):`, `docs:`, `test:` 등으로 깔끔하게 분류된 커밋 히스토리를 유지했습니다.

## 5. 바이브 코딩에서 배운 것

### 📌 핵심 인사이트

#### 1. "AI의 컨텍스트 윈도우가 곧 당신의 작업장 크기다"

AI 에이전트는 한 번에 처리할 수 있는 컨텍스트에 한계가 있습니다. `/docs/ai/STRUCTURE.md`처럼 **구조화된 싱글 소스 오브 트루스(SSOT)**를 유지하면, AI가 프로젝트 전체 맥락을 효율적으로 파악하고 일관된 코드를 생성합니다.

#### 2. "AI에게 '무엇'보다 '어떻게'와 '제약 조건'을 가르쳐라"

기능 요구사항(무엇)만 전달하면 AI는 제각기 다른 스타일의 코드를 생성합니다. `GUIDE.md`를 통해 **파일 분할 기준, 네이밍 규칙, 에러 핸들링 패턴** 등 '어떻게'를 구체적으로 정의했더니, 마치 한 사람이 작성한 것 같은 일관된 코드베이스가 만들어졌습니다.

#### 3. "하네스 문제는 시스템으로 해결한다"

AI에게 "꼼꼼하게 해줘"라고 말하는 것은 효과가 없습니다. `GEMINI.md`의 Planning Loop처럼 **검증 가능한 체크포인트**와 **물리적 제약(한 번에 하나씩)**을 시스템 레벨에서 강제해야 합니다.

#### 4. "도구별 장점을 파악하고 적재적소에 배치하라"

하나의 AI 도구에 모든 작업을 맡기지 않았습니다. Gemini CLI의 터미널 친화성(백엔드), Antigravity의 IDE 통합(프론트엔드 리팩토링), oh-my-opencode의 경량 반복(버그 수정) 등 **각 도구의 강점에 맞는 작업을 분배**함으로써 효율을 극대화했습니다.

### ⚠️ 한계와 극복

| 도전 | 해결 방법 |
|---|---|
| AI가 기존 코드를 무시하고 새로 작성 | `STRUCTURE.md`에 기존 구현을 상세 명세 |
| 다른 세션에서 이전 작업 맥락 유실 | AI가 작업 완료 시 `STRUCTURE.md` 자동 업데이트하도록 규칙화 |
| UI 디자인의 일관성 저하 | `GUIDE.md`에 타이포그래피, 컬러 계층, 아이콘 제한 규칙 명시 |
| 빌드 실패 상태에서 다음 작업 진행 | Self-Healing 루프로 통과 전까지 진행 차단 |
| 커밋 메시지 혼란 | Atomic Commit Protocol로 레이어별 분리 커밋 강제 |

## 6. 구현된 주요 UI/UX 플로우

### 🔄 사용자 여정 (User Journey)

```
최초 실행
  └─→ 온보딩 (4단계 대화형 멀티 스텝)
        ├── 닉네임 설정
        ├── 하루 시작 시간 설정
        ├── Gemini API Key 입력 & 실시간 검증
        └── 알림 설정
              └─→ 메인 뷰 진입
                    ├── Empty State → 워크스페이스 생성 유도
                    ├── 워크스페이스 생성 (3단계 스텝 모달)
                    │     ├── 기본 정보 (이름, AI 컨텍스트)
                    │     ├── 시간 관리 (코어 타임, 언플러그드 타임)
                    │     └── 고정 업무 (루틴 등록)
                    ├── 쾌속 태스크 입력 (오버레이 UX)
                    │     ├── 일반 등록 → 타임라인 자동 배정
                    │     ├── 🔥 긴급 등록 → 타임 시프트
                    │     └── 📥 인박스 보관
                    ├── 자동 상태 전환 (1초 주기 모니터링)
                    │     ├── WILL → NOW 자동 승격
                    │     └── 종료 시간 도래 → 전환 모달 자동 팝업
                    ├── 인박스 관리 (Sheet 오버레이)
                    ├── AI 회고 생성 (일/주/월 단위)
                    └── 자동 업데이트 확인 & 설치
```

### 🏗️ 기술적 하이라이트

- **프레임리스 오버레이 타이틀 바**: macOS 신호등 유지하면서 모던한 데스크톱 UI 제공
- **논리적 날짜 시스템**: `dayStartTime` 기준으로 새벽 작업자도 같은 날의 업무로 인식
- **코어 타임 시각화**: 설정된 집중 시간대에 브랜드 컬러 글로우(Glow) 효과 적용
- **스마트 라우팅**: 마감 시간 초과 시 타임라인 강제 추가 vs 인박스 이동 분기 다이얼로그
- **드래그 앤 드롭**: dnd-kit 기반 타임라인 내 태스크 순서 변경
- **i18n 완전 적용**: 한국어/영어 전면 다국어 지원, 하드코딩 문자열 제로

## 7. CI/CD & 배포 파이프라인

```
Tag Push (v*)
  └─→ GitHub Actions
        ├── macOS (aarch64-apple-darwin) — Apple Silicon
        ├── macOS (x86_64-apple-darwin)  — Intel
        └── Windows
              └─→ Tauri Action
                    ├── 프론트엔드 빌드 (npm install → vite build)
                    ├── 백엔드 컴파일 (Rust → 네이티브 바이너리)
                    ├── 아티팩트 생성 (DMG, EXE 등)
                    ├── TAURI_SIGNING_PRIVATE_KEY 서명
                    └── GitHub Release 자동 생성
```

- 앱 내 **자동 업데이트**: Tauri Updater Plugin을 통해 새 버전 감지 시 백그라운드 다운로드 및 설치
- **바이너리 서명 검증**: `TAURI_SIGNING_PUBLIC_KEY`를 통한 무결성 검증
