## 아호코라식(Aho-Corasick) 기반 고성능 비속어 필터링 라이브러리 개발

### 프로젝트 정보

- **역할** : 1인 오픈소스 라이브러리 기획 및 개발
- **기간** : 2026.01
- **기술** : Kotlin, Aho-Corasick Algorithm, Trie, Jitpack
- **한 줄 요약** : 기존 문자열 매칭의 한계를 극복하기 위해 아호코라식 알고리즘을 도입, 수천 개의 금칙어를 단 한 번의 순회로 필터링하는 고성능 라이브러리(`profanity-filter`)를 구축 및 배포.

### STAR

#### 1. Situation (기존 시스템의 한계 및 배경)

- **단순 매칭 및 정규표현식의 성능 한계:** 대다수의 비속어 필터링 로직이 단순 문자열 포함 여부나 정규표현식을 사용함. 이는 금칙어 사전의 크기가 커질수록 모든 패턴을 개별적으로 검사해야 하므로 시간 복잡도가 선형적으로 증가(O(N × K))하는 성능 저하 문제를 내포함.
- **오탐(False Positive) 및 변칙 우회 문제:** 한국어 특성상 "시1발"과 같은 변칙 표현이나, "시발점"처럼 비속어가 포함되어 있지만 실제로는 정상 단어인 경우를 구분하지 못하는 라이브러리가 많아 직접 커스텀 필터링 엔진 구축의 필요성을 느낌.

#### 2. Task (직면한 과제 및 목표)

- **기술 블로그 참고:** 배달의 민족의 기술 블로그에 나와있는 아호코라식 알고리즘을 사용한 비속어 필터에 대한 아이디어를 얻어 실제 라이브러리로 개발.
- **다중 패턴 매칭 최적화:** 수천 개의 비속어 패턴을 동시에 검색하면서도 입력 문자열의 길이에만 비례하는 시간 복잡도(O(N))를 가진 고성능 검색 엔진 구현.
- **범용 라이브러리 설계:** Java, Kotlin, Spring Boot 환경에서 모두 사용 가능하며, 사용자가 직접 금칙어를 추가/제외하거나 예외 단어(White-list)를 설정할 수 있는 유연한 API 제공.

#### 3. Action (기술적 의사결정 및 행동)

- **Trie 자료구조 구축 및 한계 분석:** 문자열 집합을 트리 형태로 관리하는 Trie를 우선 구현했으나, 공통 접두사를 반복 탐색하는 비효율을 식별. 이를 통해 대규모 패턴 매칭에 최적화된 알고리즘을 탐색함.
- **Aho-Corasick 알고리즘 도입:** Trie에 '실패 함수(Failure Function)' 개념을 결합한 아호코라식 알고리즘을 핵심 엔진으로 채택. 매칭 실패 시 루트로 돌아가는 대신 최장 공통 접미사 노드로 바로 이동하게 설계하여, 입력 문자열을 단 한 번만 순회하면서 모든 금칙어를 동시 탐색하도록 최적화.
- **오탐 방지 및 유연한 예외 처리 로직 구현:** "시발점"과 같이 비속어를 포함한 정상 단어를 필터링에서 제외할 수 있도록 `ProfanityValidator` 내부에 화이트리스트(White-list) 기반 예외 처리 로직을 설계하여 정확도를 높임.
- **오픈소스 배포 및 문서화:** Gradle 및 Maven 환경에서 즉시 도입 가능하도록 Jitpack을 통해 라이브러리를 배포하고, Java/Kotlin 각각의 사용 예시를 포함한 README를 작성하여 커뮤니티 기여 환경 구축.

#### 4. Result (정량적/정성적 성과)

- **검색 효율 극대화:** 아호코라식 도입을 통해 금칙어 개수와 상관없이 입력 문자열 길이에 비례하는 O(N)의 검색 속도를 달성, 기존 단순 Trie 방식 대비 비약적인 성능 향상.
- **실무 중심의 고도화:** 단순 알고리즘 구현을 넘어 '제외 단어 설정' 등 비즈니스 현장에서 필요한 기능들을 라이브러리에 녹여내어 실제 서비스 적용 가능한 수준의 도구로 고도화 중.
- **지식의 실체화:** 알고리즘적 이론(Trie, Aho-Corasick)을 실제 문제 해결(비속어 필터링)에 적용하여 오픈소스 라이브러리화함으로써, 기술적 깊이와 실행력을 동시에 증명.

## AI 코드 리뷰 시스템 도입

### 프로젝트 정보

- **역할** : 전체 서비스 개발
- **기간** : 2025.08 - 2025.09
- **기술** : GitLab, LM Studio, n8n
- **한 줄 요약** : 개인 스터디 시간을 활용해 사내 인프라 기반의 Zero-Budget AI 코드 리뷰 시스템을 구축하여, 리뷰 병목 해결 및 팀 코드 품질 상향 평준화 달성.

### STAR

#### 1. Situation (기존 시스템의 한계 및 배경)

- **리소스 부족과 리뷰 병목:** 차세대 국내 의약품 배송 서비스 개발의 임시 팀장으로서 실무와 매니징을 병행. 팀원들이 개발한 도메인 맥락을 파악하고 코드를 리뷰하는 데 과도한 시간과 인지적 피로 발생.
- **코드 품질 편차 우려:** 신입/인턴과 기존 팀원 간의 코드 품질 편차가 커 코드 리뷰 문화를 포기할 수 없는 상황.
- **도입 예산 제약:** 상용 AI 리뷰 SaaS(CodeRabbit 등) 도입을 고려했으나, 코드 리뷰 문화에 회의적인 경영진의 스탠스로 인해 예산 지원 불가.

#### 2. Task (직면한 과제 및 목표)

- **Zero-Budget 자동화 파이프라인 구축:** 비용 발생 없이 사내 오픈소스 인프라(GitLab, n8n, LM Studio)만으로 동작하는 AI 코드 리뷰 시스템 직접 기획 및 구현.
- **리뷰어 피로도 감소 및 품질 상향 평준화:** AI가 기본적인 로직 오류와 컨벤션을 1차 필터링하여, 휴먼 리뷰어가 핵심 비즈니스 로직과 아키텍처에만 집중할 수 있는 환경 조성.

#### 3. Action (기술적 의사결정 및 행동)

- **개인 시간을 활용한 파이프라인 구축:** 업무 외 개인 스터디 시간을 적극 할애하여, GitLab MR(Merge Request) 이벤트를 Webhook으로 수신하고 n8n 기반의 워크플로우가 즉시 트리거되도록 리뷰 프로세스 자동화 파이프라인을 직접 개발.
- **LLM 컨텍스트 최적화 (Noise Filtering):** 거대한 MR Diff 전체 전달 시 발생하는 토큰 낭비 및 환각(Hallucination) 방지. 변경 사항을 Hunk 단위로 분해하고, Import 문이나 주석 등 로직과 무관한 노이즈를 필터링하여 프롬프트 페이로드 정제.
- **추론 환경 안정화 및 부하 분산:** 로컬 추론 환경에 코드 분석 특화 모델(`gpt-oss-20b`) 적용. 다수 MR 동시 발생 시 병목 해결을 위해 라운드 로빈(Round-Robin) 방식으로 AI 요청을 분산 처리하여 응답 안정성과 전체 처리량 개선.

#### 4. Result (정량적/정성적 성과)

- **개발 생산성 및 코드 품질 향상:** 1차 AI 리뷰의 즉각적인 피드백으로 신입 팀원의 코드 품질 상향 평준화 달성. 리뷰 소요 시간 단축 및 피로도 감소로 전체 기능 배포 속도 향상.
- **경영진 설득 및 성과 인정:** 개인 시간을 투자해 주도적으로 팀의 생산성 병목을 해결하고 코드 품질을 개선한 지표(에러 리포트)를 증명하여, 실제 성과 평가 및 연봉 인상에 긍정적으로 반영됨.

## AI 시스템 프롬프트 생성 서비스 개발

### 프로젝트 정보

- **역할** : 백엔드 개발
- **기간** : 2025.08 - 2025.10
- **기술** : Kotlin, Spring Boot, MySQL, Google AI Studio API, OpenAI API
- **한 줄 요약** : 코루틴 무한 생성으로 인한 외부 API Rate Limit 초과 문제를 Channel 기반 Worker Pool과 Fan-out 구조로 해결하여, 대규모 동시 요청 환경의 서비스 안정성 및 처리 효율 극대화 달성.

### STAR

#### 1. Situation (기존 시스템의 한계 및 배경)

- **Unbounded Concurrency로 인한 에러 발생:** 다중 사용자 동시 요청 시, 매 요청마다 `CoroutineScope.launch`로 새로운 워커를 생성(Fire-and-Forget)하는 구조를 사용. 이로 인해 순식간에 외부 API(OpenAI, Google AI Studio)로 트래픽이 몰려 엄격한 RPM(Requests Per Minute) 제한을 초과(429 에러)하는 생성 실패 이슈가 반복적으로 발생.
- **파편화된 상태 추적의 한계:** 수많은 코루틴이 개별적으로 실행되면서 라이프사이클 관리가 어려워짐. 이로 인해 데이터셋 생성 및 프롬프트 테스트 등 여러 단계를 거치는 긴 호흡의 작업 진행 상태를 프론트엔드에 실시간으로 전달하는 데 구조적 어려움 존재.

#### 2. Task (직면한 과제 및 목표)

- **동시성 제어 및 병목 해결:** 외부 AI API의 Rate Limit을 준수하면서도 여러 사용자의 요청을 안전하게 병렬 처리할 수 있는 제어 가능한 동시성(Bounded Concurrency) 아키텍처 설계 및 구현.
- **실시간 모니터링 환경 구축:** 복잡한 비동기 작업 파이프라인 속에서도 각 요청의 진행 상태를 성능 저하 없이 추적하여 프론트엔드에서 모니터링할 수 있도록 상태 관리 로직 최적화.

#### 3. Action (기술적 의사결정 및 행동)

- **Worker Pool + Fan-out 아키텍처 설계:** 무분별한 `launch` 생성을 제거하고, Kotlin `Channel`을 큐(버퍼)로 활용하여 Backpressure(배압) 메커니즘을 적용. 고정된 수의 코루틴 워커(Worker Pool)만이 Channel에서 요청을 가져가 처리(Fan-out)하도록 구현하여 동시 API 호출량을 물리적으로 통제.
- **Lock-Free 기반 상태 관리 로직 구현:** 여러 코루틴 워커가 동시에 상태를 업데이트할 때 발생하는 Race Condition을 방지하기 위해 `ConcurrentHashMap`을 도입. 무거운 동기화(Synchronized) 블록 없이 스레드 안전(Thread-safe)하게 작업 진행 현황을 관리하여 모니터링 성능 저하를 방지.

#### 4. Result (정량적/정성적 성과)

- **생성 실패율 완화 및 시스템 안정성 확보:** 트래픽 스파이크 발생 시에도 Worker Pool을 통해 요청이 시스템이 감당 가능한 속도로만 소비되도록 하여 API 호출 실패 이슈를 해결하고 전체 처리 효율을 대폭 개선.
- **사용자 경험(UX) 개선:** 병목 구간 제거 및 고성능 상태 동기화 로직을 통해, 프론트엔드에서 데이터셋 생성부터 프롬프트 테스트까지의 전 과정을 실시간으로 투명하게 모니터링할 수 있는 안정적인 환경 제공.

## 차세대 국내 의약품 배송 서비스 개발 및 레거시 전환

### 프로젝트 정보

- **역할** : 프로젝트 개발 팀장 (PL) / 도메인 설계 및 마이그레이션 파이프라인 구축
- **기간** : 2025.07 - 2025.08
- **기술** : Kotlin, Spring Boot, RDBMS
- **한 줄 요약** : 타 팀으로부터 이관받은 복잡한 배송 프로젝트의 도메인 지식을 체계화하고 팀 간 컨벤션을 수립하여 협업 효율을 개선했으며, Kotlin `sealed class` 기반의 Event Sourcing 아키텍처를 도입하여 레거시 전환을 성공적으로 완수함.

### STAR

#### 1. Situation (기존 시스템의 한계 및 배경)

- **도메인 이해도 부족 및 스크럼 지연:** 타 팀에서 초기 세팅 및 개발을 진행하던 프로젝트를 긴급하게 이관받아 투입됨. 복잡하고 고객사별로 상이한 배송 도메인 정책을 충분히 소화하지 못한 상태로 스크럼을 진행하여, 초기 마일스톤 기한을 준수하지 못하는 병목 이슈 발생.
- **코드 디자인 파편화:** 타 팀과 자사 팀 간의 파일 스트럭처 및 코드 아키텍처 룰이 달라, 이관된 코드의 버그 추적 및 유지보수 과정에서 심각한 개발 피로도와 리소스 낭비 발생.
- **레거시 상태 관리의 한계:** 기존 레거시 시스템은 배송 상태의 변경 이력을 추적하기 어려운 구조적 결함이 있어, 신규 시스템으로의 안전한 마이그레이션 및 확장에 제약이 존재함.

#### 2. Task (직면한 과제 및 목표)

- **도메인 온보딩 및 개발 프로세스 정상화:** 파편화된 도메인 지식을 단기간에 체계화하여 팀원들의 이해도를 높이고, 팀 간 이질적인 개발 환경을 통합할 수 있는 표준 컨벤션 수립.
- **레거시 데이터 마이그레이션 및 아키텍처 재설계:** 상태 변경 이력을 완벽하게 추적하고 복원할 수 있는 도메인 모델링을 새롭게 설계하고, 약 5천 건 이상의 레거시 데이터를 메모리 이슈(OOM) 없이 안전하게 이관.

#### 3. Action (기술적 의사결정 및 행동)

- **Ubiquitous Language 기반 위키 구축 (실패를 통한 개선):** 초기 도메인 분석 부족으로 인한 일정 지연 경험을 반면교사 삼아, 기획팀과 주도적으로 회의를 주최. 고객사별 상이한 도메인 흐름과 정책을 유비쿼터스 언어(Ubiquitous Language)로 정의하고 사내 위키를 구축하여 팀원들의 온보딩 효율을 극대화함.

## 사내 프레임워크 멀티바이트 문자(UTF-8) 스트리밍 디코딩 장애 해결

### 프로젝트 정보

- **역할** : 백엔드 프레임워크 트러블슈팅 및 로직 개선
- **기간** : 2025.04
- **기술** : Kotlin, Spring Boot (WebFlux), Netty
- **한 줄 요약** : WebFlux(Netty)의 8KB 청크 스트리밍 환경에서 발생하는 UTF-8 멀티바이트 문자 잘림(U+FFFD) 장애를 비트 연산 기반의 바이트 경계 탐색 알고리즘으로 해결하여 데이터 무결성 확보.

### STAR

#### 1. Situation (기존 시스템의 한계 및 배경)

- **대용량 페이로드 인코딩 장애:** WebFlux 기반 사내 프레임워크를 통해 서버로 데이터를 전송할 때, 특정 상황(약 2,700자 이상의 한글 또는 이모지 포함)에서 문자가 ``(U+FFFD)로 깨지는 버그가 제보됨.
- **스트리밍 환경의 구조적 문제 발견:** 원인 분석 결과, Netty가 대량의 요청 데이터를 기본 8KB(8192바이트) 단위의 청크(Chunk)로 나누어 수신하는 과정에서 원인이 있음을 식별. 한글(3바이트)의 경우 8192바이트 경계에서 2730번째 글자가 2바이트와 1바이트로 분할 수신되며, 이를 청크 단위로 단순 디코딩하여 누적(Concatenate)할 때 유효하지 않은 시퀀스로 간주되어 디코딩이 깨지는 현상 확인.

#### 2. Task (직면한 과제 및 목표)

- **데이터 무결성 확보:** 청크 경계에 걸친 가변 길이(1~4바이트) UTF-8 멀티바이트 문자가 깨지지 않도록 디코딩 로직 개선.
- **WebFlux 스트리밍 이점 유지:** 전체 요청 Body를 메모리에 한 번에 적재(Aggregating)하여 해결하는 쉬운 우회법 대신, 메모리 효율성과 논블로킹 특성을 유지하기 위해 청크 단위 스트리밍을 유지하면서 경계 문제를 해결하는 최적화된 알고리즘 설계.

#### 3. Action (기술적 의사결정 및 행동)

- **비트 연산 기반 UTF-8 경계 탐색 알고리즘 개발:** 수신된 청크의 `ByteArray` 끝에서부터 역방향으로 순회하며 비트 연산(Bitwise Operation)을 수행하는 `findLastCompleteUtf8CharBoundary` 로직을 직접 구현. 바이트의 헤더(예: `110xxxxx`, `1110xxxx`, `11110xxx`)를 판별하여 완전한 문자의 시퀀스가 끝나는 정확한 인덱스를 계산함.
- **불완전 바이트 이관(Carry-over) 처리:** 탐색한 경계 인덱스까지만 안전하게 문자열로 디코딩하여 누적하고, 경계에 걸쳐 잘려나간 나머지 바이트(1~3바이트)는 버리지 않고 다음 청크 수신 시 앞부분에 연결(Slice & Carry-over)하여 완전한 문자로 조립되도록 파이프라인 재설계.

#### 4. Result (정량적/정성적 성과)

- **장애 원천 차단 및 안정성 향상:** 대규모 텍스트 스트리밍 환경에서도 한글, 이모지 등 멀티바이트 문자의 서로게이트 페어 깨짐 및 데이터 손실 문제를 100% 해결.
- **Low-Level 문제 해결 역량 증명:** 단순한 프레임워크 기능 사용을 넘어, 인코딩 스펙(UTF-8 바이트 구조)과 논블로킹 네트워크 계층(Netty Chunking)의 동작 원리에 대한 깊은 이해를 바탕으로 프레임워크 레벨의 코어 로직을 안정적으로 개선함.

## 레거시 국내 의약품 배송 서비스 조회 API 성능 최적화 (장애 대응)

### 프로젝트 정보

- **역할** : 백엔드 성능 최적화 및 레거시 리팩터링
- **기간** : 2025.02 - 2025.02
- **기술** : Kotlin, Spring Boot, RDBMS
- **한 줄 요약** : Depth Level 4 이상의 복잡한 중첩 구조에서 발생한 N+1 쿼리 병목을 Batch Fetching과 인메모리 매핑 구조로 리팩터링하여 API 응답 시간을 90% 이상 개선하고 타임아웃 장애를 해결함.

### STAR

#### 1. Situation (기존 시스템의 한계 및 배경)

- **조회 API 타임아웃 장애 발생:** 레거시 배차 완료 목록 조회 API 호출 시, 데이터 증가에 따라 응답 시간이 3~5초 이상 소요되며 프론트엔드 타임아웃(Timeout)이 빈번하게 발생하는 서비스 장애 수준의 지연 발생.
- **N+1 쿼리로 인한 DB 부하:** 원인 분석 결과, 배송(Delivery) 건에 포함된 패키지(Package)의 상태 로그 및 시간 정보를 조회할 때 반복문(forEach) 내부에서 개별 SQL 쿼리가 실행되는 구조적 결함 확인.
- **선형적으로 증가하는 라운드트립 (O(N)):** 50개의 배송 건 조회 시 100회 이상의 추가 쿼리가 발생하는 등, 조회 건수에 비례하여 DB 라운드트립이 선형적으로 증가하는 비효율적인 로직이 병목의 핵심 원인으로 식별됨.

#### 2. Task (직면한 과제 및 목표)

- **응답 시간 단축 및 병목 제거:** DB 라운드트립 횟수를 근본적으로 줄여 API 응답 시간을 1초 미만(Sub-second)의 안정적인 수치로 최적화.
- **안정적인 데이터 패칭 구조 설계:** 복잡한 중첩 엔티티 구조(Depth Level 4 이상)를 유지하면서도 대량의 데이터를 부하 없이 조회할 수 있는 최적화된 데이터 로드 패턴(Data Load Pattern) 도입.

#### 3. Action (기술적 의사결정 및 행동)

- **Batch Fetching 및 `IN` 절을 활용한 일괄 조회 도입:** 반복문 내의 단건 쿼리를 제거하고, 조회 대상 패키지 ID 리스트를 선제적으로 수집. 이후 수집된 ID 리스트를 SQL의 `IN` 절 파라미터로 넘겨 연관 데이터를 단일 쿼리로 한 번에 조회하도록 Batch Fetching 전략 적용.
- **애플리케이션 레벨의 In-memory Mapping 구현:** DB에서 일괄 조회한 평면적인(Flat) 결과 데이터를 애플리케이션 메모리 상에서 `Map` 자료구조로 변환. 이후 컬렉션의 `groupBy` 및 `associateBy` 등을 활용해 각 배송/패키지 객체에 알맞게 매핑함으로써 DB 연산 부하를 애플리케이션 계층으로 안전하게 분산.
- **모니터링 기반 검증:** `DurationUtil`을 활용해 쿼리 실행 구간 및 인메모리 조립 구간의 실행 시간을 정밀 측정하여, O(N)이었던 쿼리 호출 횟수가 상수(O(1)) 수준으로 고정되었음을 데이터로 검증 후 코드 반영.

#### 4. Result (정량적/정성적 성과)

- **API 응답 속도 90% 이상 개선:** 기존 최대 5초 이상 소요되던 지연 시간을 0.2 ~ 0.5초 미만으로 대폭 단축하여 사용자 대기 시간을 획기적으로 개선.
- **서비스 안정성 확보:** DB 연결 고갈 및 프론트엔드 타임아웃 문제를 원천적으로 해결하여 대용량 목록 조회 시에도 일관된 응답 품질(QoS)과 운영 안정성을 확보함.

- **순차적 네트워크 워터폴(Waterfall) 현상:** 1번 데이터를 받아야만 2번, 3번 데이터를 순차적으로 호출할 수 있는 프론트엔드 종속적인 데이터 관계망이 형성됨. 결과적으로 로컬 환경에서조차 페이지 로딩에 2~3초가 소요되는 심각한 지연 발생.
- **커뮤니케이션 병목:** 부서 간의 소통 부재와 의견 충돌로 인해 구두상으로 제안한 API 구조 개선안이 지속적으로 반려되며, 비효율적인 레거시 구조가 강제되는 상황.

#### 2. Task (직면한 과제 및 목표)

- **데이터 기반의 설득 및 프로세스 정상화:** 감정적 충돌이나 구두 설득을 넘어, 아키텍처 변경의 효용성을 객관적인 수치와 시각적 결과물로 증명하여 기획팀과 이해관계자를 설득.
- **BFF 아키텍처 도입 및 API 최적화:** 프론트엔드에 흩어져 있는 애그리게이트(Aggregate) 단위의 데이터 조합 로직을 백엔드로 이관하여, 단일 호출만으로 화면 렌더링에 필요한 데이터만 반환하는 효율적인 구조 설계.

#### 3. Action (기술적 의사결정 및 행동)

- **Show, Don't Tell (프로토타입 시연을 통한 설득):** 개선된 아키텍처를 적용한 로컬 환경의 프로토타입을 직접 구현. 기존 2~3초가 걸리던 로딩 속도가 즉각적으로 렌더링되는 결과와 최적화된 네트워크 페이로드를 이해관계자들에게 직접 시연하여 아키텍처 변경의 당위성을 기술적으로 증명하고 채택을 이끌어냄.
- **BFF(Backend-for-Frontend) 패턴 적용:** 프론트엔드의 뷰(View) 요구사항에 정확히 맞춘 전용 API 레이어를 구축. 여러 도메인 서비스의 데이터를 백엔드 서버에서 병렬로 호출하여 조립(Aggregation)한 뒤, 불필요한 속성을 제거한 DTO로 변환하여 클라이언트에 제공.
- **End-to-End 주도:** 기획 회의 참여부터 ERD 설계, 시스템 아키텍처 구상, 실제 비즈니스 로직 구현까지 전 과정을 주도하며 데이터 정합성 강화 및 네트워크 왕복 횟수(Round-trip) 최소화 로직을 반영.

#### 4. Result (정량적/정성적 성과)

- **극적인 성능 개선:** 불필요한 네트워크 왕복을 제거하고 백엔드에서 데이터를 집계한 결과, 서비스 평균 응답 시간을 **2.56초에서 314ms로 약 87% 이상 대폭 개선**하여 사용자 경험(UX)을 극대화함.
- **아키텍처 표준화 및 팀 시너지 창출:** 프론트엔드의 복잡한 상태 관리 로직을 백엔드로 이관하여 클라이언트 개발 생산성을 높였으며, 기술적 근거를 바탕으로 타 부서와의 협업 신뢰도를 회복하고 효율적인 개발 문화를 정착시킴.

## 레거시 해상 운송 플랫폼 이메일 발송 성능 최적화

### 프로젝트 정보

- **역할** : 백엔드 기능 유지보수 및 성능 최적화 (인턴)
- **기간** : 2024.03 - 2024.03
- **기술** : Kotlin, Spring Boot, Coroutines
- **한 줄 요약** : 문제 해결을 위해 외부 코루틴 스터디를 직접 조직 및 운영하며 기술을 습득했고, 이를 바탕으로 동기적 이메일 발송 로직을 비동기 병렬 처리로 리팩터링하여 API 응답 시간을 80% 이상 단축함.

### STAR

#### 1. Situation (기존 시스템의 한계 및 배경)

- **순차 처리로 인한 심각한 병목 현상:** 해상 운송을 진행하는 화주/포워더 매칭 플랫폼에서, 화주가 견적을 요청하면 다수의 포워더에게 알림 이메일이 발송되는 기능 존재.
- **I/O 블로킹 및 응답 지연:** 기존 로직은 반복문(`forEach`)을 통해 이메일을 순차적(Sequential)으로 발송하고 있었음. 외부 SMTP 서버와 통신하는 I/O 작업이 동기적으로 블로킹되어, 포워더 계정이 10개만 되어도 견적 요청 API 응답에 최대 60초가 소요됨.
- **UX 저해 및 타임아웃 위험:** 화주 입장에서는 견적 요청 버튼 클릭 후 화면이 멈춘 것처럼 느껴지며, 네트워크 환경에 따라 타임아웃(Timeout)이 발생할 위험이 큰 상태.

#### 2. Task (직면한 과제 및 목표)

- **I/O 대기 시간 최소화:** 이메일 발송이라는 대표적인 I/O Bound 작업에서 발생하는 대기 시간을 줄여 전체 API 응답 속도 개선.
- **기술적 한계 극복:** 당시 실무적으로 코루틴(Coroutines) 활용 경험이 부재했으나, 스레드 풀 고갈 없이 비동기 처리를 구현하기 위해 코틀린의 Idiomatic한 동시성 도구를 빠르게 학습하고 실무에 적용해야 했음.

#### 3. Action (기술적 의사결정 및 행동)

- **주도적인 기술 학습 및 스터디 운영:** 비동기 프로그래밍의 필요성을 인지한 후, 인프런(Inflearn) 커뮤니티를 통해 '코틀린 코루틴 스터디'를 직접 개설하고 외부 개발자들을 모집했습니다. 스터디를 리딩하며 코루틴의 동작 원리와 I/O 최적화 기법을 깊이 있게 학습하고 이를 즉시 실무 리팩터링에 적용했습니다.
- **Coroutine `async` / `awaitAll` 구조 리팩터링:** 기존의 단일 스레드 순차 처리 루프를 분해하고, 여러 개의 이메일 발송 작업을 `async` 블록으로 감싸 병렬로 실행되도록 구조를 재설계했습니다. 이후 `awaitAll()`을 통해 모든 비동기 작업이 안전하게 완료됨을 보장한 뒤 클라이언트에게 응답을 반환하도록 흐름을 제어했습니다.

#### 4. Result (정량적/정성적 성과)

- **응답 속도 80% 이상 개선:** 10건 기준 최대 60초까지 지연되던 API 응답 시간을 8~11초로 대폭 단축하여 극적인 성능 향상을 이끌어냄.
- **자기 주도적 문제 해결 역량 입증:** 당면한 기술적 과제를 회피하지 않고 외부 스터디 조직이라는 적극적인 방식으로 학습 곡선(Learning Curve)을 극복했으며, 이를 실제 서비스 성능 개선이라는 정량적 성과로 증명함.
